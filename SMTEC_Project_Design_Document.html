SMTEC Student Management System

Product Design & Technical Specification Document
Version 1.0 Status: Approved for Development Target: Web Platform

1. Executive Summary

The SMTEC Student Management System is a comprehensive web application designed to digitize and streamline the academic and co-curricular management of St. Mother Theresa Engineering College. The system serves two primary user groups: Administrators/Faculty and Students.

It aims to replace manual record-keeping with a centralized, real-time platform for managing student data, tracking skill development, monitoring academic results, and fostering a competitive learning environment through gamified leaderboards.

2. Technology Stack

The project will be built using a modern, scalable, and performance-oriented stack.

Component Technology Rationale
Frontend Framework React.js + Vite Component-based architecture, fast build times, and rich ecosystem.
Styling Tailwind CSS Utility-first CSS for rapid UI development, responsiveness, and dark mode support.
Backend / Database Firebase (Firestore & Auth) Real-time database, serverless authentication, and easy scaling.
Icons Lucide React Clean, consistent, and lightweight SVG icons.
Charts Recharts Composable charting library for data visualization.
Data Handling XLSX For parsing and generating Excel files for bulk data operations.

3. User Roles & Permissions

Role Access Level Key Responsibilities
Admin / HOD Full System Access
  • Manage student records (CRUD).
  • Upload bulk data (Excel/Google Sheets).
  • Manage academic batches (Promotion/Graduation).
  • Assign skill points and view reports.
  • Access Alumni directory.
Student Restricted Personal Access
  • View personal profile and academic stats.
  • View skill reports and charts.
  • Access academic results.
  • View global leaderboard.

4. Detailed Feature Specifications

4.1. Authentication Module

  • Login Page: Secure login using Email/Password.
  • Role Detection: Automatically route users to /admin or /student based on their Firestore role.
  • Security: Protected routes (PrivateRoute component) to prevent unauthorized access.

4.2. Common UI Components

  • Sidebar Navigation: Responsive sidebar with collapsible menu for mobile. Contains navigation links specific to the user role.
  • Header: Sticky header with page title, theme toggle (Dark/Light mode), notification bell, and user profile summary.
  • Theme System: Full dark mode support using Tailwind's dark: variant.

4.3. Admin Module

A. Dashboard

Path: /admin

  • Overview Cards: Display total students, total departments, and active batches.
  • Quick Actions: Shortcuts to Add Student, Upload Data, etc.

B. Student Management

Path: /admin/students

  • Student List: Tabular view of students with pagination/filtering.
  • Filtering: Filter by Department (CSE, ECE, etc.) and Year (1-4).
  • Add/Edit Student: Modal form to add or update student details (Name, RegNo, Email, Dept, Year, Sem).
  • Delete: Option to remove student records.
  • Department Icons: Use specific icons for each department (e.g., Code for CSE, Globe for IT).

C. Data Upload & Sync

Path: /admin/upload

  • Excel Upload: Drag-and-drop interface to upload .xlsx files for bulk student creation.
  • Google Sheets Sync: Integration to fetch live data from a Google Sheet URL.
  • Data Validation: Check for duplicate Register Numbers before insertion.

D. Skill Management

Path: /admin/skills

  • Skill Entry: Form to award points to students for activities (Paper Presentation, Sports, etc.).
  • Skill Reports: Hierarchical view (Dept -> Year -> Student) to view individual student progress.

E. Batch Management

Path: /admin/batches

  • Promotion: One-click promotion of students to the next academic year (e.g., Year 1 -> Year 2).
  • Graduation: Move Final Year students to the Alumni collection.

F. Alumni Directory

Path: /admin/alumni

  • Archive View: Browse graduated students by Batch Year and Department.
  • Search: Find alumni by name or register number.

4.4. Student Module

A. Overview (Dashboard)

Path: /student

  • Welcome Banner: Personalized greeting with campus background image.
  • Quick Stats: Cards showing Total Skill Points and Activities Logged.
  • Recent Achievements: List of the 3 most recent activities.

B. Skill Report

Path: /student/skills

  • Charts: Visual representation (Bar/Line chart) of skill points over time (Daily, Weekly, Monthly).
  • Activity Log: Detailed list of all participation with dates and points awarded.

C. Leaderboard

Path: /student/leaderboard (Also available to Admin)

  • Global Ranking: Rank students based on total skill points.
  • Filters: Filter leaderboard by Department and Year.
  • "Me" Highlight: Highlight the current user's row in the leaderboard.
  • Top 3 Styling: Special Gold/Silver/Bronze styling for top rankers.

5. Database Schema (Firestore)

The application uses a NoSQL document structure.

Collection: users

Stores authentication and profile data for all users.

{
  "uid": "string (Auth ID)",
  "name": "string",
  "email": "string",
  "role": "string ('admin' | 'student')",
  "regNo": "string (Unique)",
  "dept": "string",
  "year": number,
  "semester": number,
  "createdAt": timestamp
}

Collection: student_skills

Stores individual activity records.

{
  "id": "string (Auto-ID)",
  "regNo": "string (Foreign Key)",
  "studentName": "string",
  "skillName": "string",
  "category": "string",
  "points": number,
  "date": timestamp,
  "semester": number,
  "year": number
}

Collection: alumni

Stores records of graduated students.

{
  "id": "string",
  "name": "string",
  "batch": "string (e.g., '2021-2025')",
  "dept": "string",
  ...studentData
}

6. UI/UX Design Guidelines

  • Color Palette:
    • Primary: Blue (Tailwind blue-600)
    • Background: Slate/Gray (gray-50 / gray-900 for dark mode)
    • Accents: Yellow/Gold for achievements.
  • Typography: Sans-serif fonts (Inter or system default) for readability.
  • Responsiveness: Mobile-first approach.
    • Grid layouts should collapse to 1 column on mobile (grid-cols-1) and expand on desktop.
    • Sidebar should be a drawer on mobile and fixed on desktop.
    • Tables should be scrollable horizontally on small screens.
  • Interactions:
    • Hover effects on cards and buttons.
    • Smooth transitions for theme switching and page loads.
    • Loading spinners for async operations.

  5. Phase 5: Polish

    • Mobile responsiveness testing.
    • Final asset integration (Logos, Campus images).
